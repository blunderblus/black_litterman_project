## 맥락(Context)

BL은 여러 출처의 법인 데이터를 결합한다. 각 출처는 서로 다른 식별자를 사용한다.

- **corp_code**: DART(전자공시) 고유 기업코드. **8자리**(예 `00126380`).
- **biz_reg_no(사업자등록번호)**: 국세청 체계의 사업자 식별자. **10자리** `XXX-XX-XXXXX`.
- **jurir_no(법인등록번호)**: 등기 법인 식별자. **13자리** `XXXXXX-XXXXXXX`.
- **stock_code**: 상장사 종목코드(상장사만 존재). **6자리**(예 `005930`).

세분화 축인 Tier(T1 상장 외감 / T2 비상장 중소 / T3 가상 섹터 노드)와 Track(A 매크로 / B Naver 뉴스) 데이터를 동일 법인으로 묶으려면 이 식별자들 사이의 정확한 매핑이 필수다.

과거 토이 프로젝트에서 **가장 치명적인 데이터 버그**가 여기서 발생했다.

- **As-is 실제 결함**: biz_reg_no(사업자번호)를 jurir_no(법인번호)와 **잘못 조인**했다. 두 번호는 체계도 자릿수도 다른 별개의 식별자(10자리 vs 13자리)인데 직접 등호 조인으로 묶은 결과, **약 99.4%의 데이터가 tier=UNKNOWN으로 소실**되었다(토이 기준 **추정치, 미검증** — 목표치 아님, [02 §주의](../02-data-pipeline.md) 참조). 즉 유니버스 대부분이 세분화·뷰 결합에서 탈락하여 BL 입력 자체가 빈약해졌다.
- **결함의 직접 증거(smoking gun)**: 실제 산출물 `bl_input_details_T1_202510.csv`의 `biz_reg_no`(10자리 칼럼)에 jurir_no류 **13자리 값**(예 `1101112248882`)이 혼입되어 있다. 사업자번호 칼럼에 법인번호류 값이 들어간 것은 이종 식별자 오조인의 직접 흔적이다([02 §4.1](../02-data-pipeline.md) 참조).
- 수집 노트북(stage 01)은 DART `corpCode.xml`(고유번호 파일)로 canonical `corp_code`를 확보하고 `company.json`(기업개황)에서 DART 원시필드 `bizr_no`(=표준 컬럼 `biz_reg_no`로 정규화)·`jurir_no`를 보강하고 있었다. 그러나 이들을 정합적으로 잇는 단일 진실원천(single source of truth)이 없었고, 조인이 코드 곳곳에 암묵적으로 흩어져 동일한 오조인이 재발할 구조였다.

> 표기 정규화: DART 원시필드 `bizr_no`는 본 프로젝트 표준 컬럼명 `biz_reg_no`로 정규화하며, 이후 본 문서는 `biz_reg_no`로 통일 표기한다(02 문서에서 TARGET_MASTER 원천은 `bizr_no`, ID_CROSSWALK는 `biz_reg_no` 컬럼명을 사용).

격상판은 데이터 손실 없는 정확한 결합과, 조인 정합성의 검증 가능성을 요구한다.

## 결정(Decision)

**명시적 ID crosswalk(식별자 매핑 테이블)** 를 단일 진실원천으로 도입한다.

- DuckDB에 `ID_CROSSWALK` 테이블을 둔다. 컬럼(예): `corp_code`(주키 PK, DART 고유, **VARCHAR(8)**, 예 `00126380`), `biz_reg_no`(VARCHAR(10)), `jurir_no`(VARCHAR(13)), `stock_code`(VARCHAR(6)), `is_listed`, `tier`, `is_virtual`(T3 가상 노드 여부), `source`, `as_of_date`, `confidence`. 전체 스키마는 [02 §4.2.1](../02-data-pipeline.md)이 권위 소스다.
- **내부 표준 키(canonical key)** 는 `corp_code`로 고정한다. 모든 출처 데이터는 적재 시 crosswalk를 통해 `corp_code`로 정규화한 뒤 결합한다. 출처별 식별자 간 **직접 조인을 금지**하고, 반드시 crosswalk를 경유한다.
- crosswalk는 **결정적·검증된 매핑만** 채운다. DART 원천(corp_code↔biz_reg_no↔jurir_no↔stock_code 동시 보유)을 1차 근거로 사용하고, 매핑 출처와 신뢰도(confidence)를 기록한다.
- **무결성 제약·품질 게이트**를 둔다([02 §4.3](../02-data-pipeline.md) 권위).
  - **형식 검증**: 자릿수·체크섬을 식별자별로 분리 적용한다 — `corp_code` 8자리(VARCHAR(8), 형식), `biz_reg_no` 10자리(VARCHAR(10), 국세청 사업자번호 체크섬), `jurir_no` 13자리(VARCHAR(13), 형식·체크 digit), `stock_code` 6자리(상장사 한정). 체크섬 규칙은 식별자마다 상이하므로 일괄 적용하지 않는다. "형식이 다른 번호를 같은 칼럼에 넣는" 오류(99.4% 소실의 진원)를 차단한다.
  - 1:1 매핑 위반(중복 corp_code, 한 biz_reg_no가 다수 corp_code에 매핑 등) 탐지.
  - **매핑 커버리지 게이트**: crosswalk를 통해 tier가 부여된 비율을 측정하고, `UNKNOWN` 비율이 임계치를 넘으면 빌드를 실패시킨다(과거 99.4% 소실 재발 방지의 회귀 가드). **임계치는 초기 캘리브레이션 후 확정**하며([02 §4.3·§주의](../02-data-pipeline.md)), 확정 전까지는 게이트를 경고(warn)로 운용한다. 정량 합격선은 02(권위 소스)에서 단일 확정하고, [03 로드맵 P2 DoD](../../planning/03-roadmap.md)·[02 PRD §3.2 AC](../../planning/02-prd.md)는 그 값을 참조한다.
- 미매핑 엔티티는 조용히 버리지 않고 `UNKNOWN`으로 명시 보존하여 추적·보정 대상으로 남긴다(silent drop 금지).

## 근거(Rationale)

서로 다른 식별자 체계를 직접 조인하는 것은 정의상 불가능에 가깝고(자릿수·체계 상이), 한 번 잘못되면 대량 silent loss로 이어진다. 매핑을 코드에 흩어두는 대신 **단일 테이블 + 무결성 제약 + 커버리지 게이트**로 외부화하면, 정합성이 데이터로 보증되고 회귀 테스트가 가능해진다.

대안 비교(검토한 세 대안을 한 표로 통합):

| 기준 | 명시적 ID crosswalk (채택) | 직접 조인 유지 | 이름(상호) 매칭 | 결합 안 함(사일로) |
|---|---|---|---|---|
| 정확성(오조인 위험) | 높음(검증된 매핑만) | 매우 낮음(이종 키 직접조인) | 낮음(동명이인·표기변형) | 해당 없음(결합 자체 없음) |
| 데이터 손실 방지 | 높음(UNKNOWN 보존+게이트) | 낮음(과거 99.4% 소실, 추정·미검증) | 중간 | 손실은 없으나 결합 부재 |
| 검증·회귀 가능성 | 높음(제약·커버리지 게이트) | 낮음(암묵 조인 산재) | 낮음 | 해당 없음 |
| 단일 진실원천 | 있음 | 없음 | 없음 | 없음 |
| 구현·유지 비용 | 중간(테이블·게이트 구축) | 낮음(겉보기만) | 높음(정규화·퍼지매칭) | 낮음 |
| 신규 출처 추가 용이성 | 높음(crosswalk만 확장) | 낮음 | 낮음 | 해당 없음 |
| 시스템 목적 부합 | 부합 | 부합(겉보기) | 부분 | **불충족(4축 뷰·Tier 불가)** |

## 결과(Consequences)

긍정적:
- biz_reg_no↔jurir_no류 오조인과 대량 silent loss를 구조적으로 차단. 유니버스 커버리지가 정상화되어 BL 입력의 빈약화 문제 해소.
- corp_code 단일 canonical key로 모든 Track·Tier 데이터 결합이 일관되고, 신규 데이터 출처 추가가 crosswalk 확장만으로 가능.
- 커버리지 게이트가 회귀 가드로 작동하여 동일 결함의 재발을 빌드 단계에서 차단.

부정적:
- crosswalk 구축·유지(매핑 수집, 형식·1:1 검증, as_of 갱신) 운영 부담이 생긴다.
- 매핑 근거가 부족한 비상장·중소 법인(T2)은 일부 미매핑이 남을 수 있어, UNKNOWN 보정 절차가 지속 필요.
- 모든 조인이 crosswalk 경유로 한 단계 늘어 쿼리가 다소 복잡해진다(성능은 DuckDB로 충분).

후속작업:
- [ ] `ID_CROSSWALK` 스키마·무결성 제약(형식·1:1·외래키) 정의 및 DART 원천 기반 초기 적재(스키마는 [02 §4.2.1](../02-data-pipeline.md) 권위).
- [ ] 식별자 형식·체크섬 검증 함수와 1:1 위반 탐지 점검 구현(식별자별 체크섬 규칙 분리).
- [ ] tier 부여 커버리지 측정 + UNKNOWN 임계치 빌드 게이트(회귀 테스트) 도입. 임계치 초기값은 02 캘리브레이션으로 확정.
- [ ] **직접조인 금지 규약은 [02 §4.2](../02-data-pipeline.md)에 이미 명문화됨** — 잔여 작업은 각 단계 코드에 남은 출처별 직접 조인을 crosswalk 경유로 리팩터링하는 것.
- [ ] 이름매칭은 후보 생성 보조로만 사용하는 절차·임계치 정의(최종 키 불가).

## 대안(Considered Alternatives)

기각 사유만 짧게 남긴다(상세 비교는 위 [근거](#근거rationale) 표로 위임).

- **직접 조인 유지**: 이종 식별자 직접 조인이 과거 99.4%(토이 기준 추정·미검증) 데이터 소실의 직접 원인. 정확성·손실 방지·검증성 모두 열위로 **기각**.
- **이름(상호) 매칭으로 결합**: 한국 법인명은 표기 변형·동명·계열 중복이 많아 정밀도가 낮고 그대로 키로 쓰면 또 다른 오결합을 부른다. crosswalk **후보 생성 보조 용도로만 한정 채택**, 최종 키로는 기각.
- **출처별 키를 그대로 유지하고 결합하지 않음(사일로)**: 데이터 사일로화로 4축 뷰 앙상블·Tier 세분화가 불가능해져 시스템 목적과 충돌하므로 **기각**.
